Architecture Overview
Clean Architecture · Domain-Driven Design · CQRS · Mediator Pattern

The DeveloperStore solution is built on Clean Architecture combined with Domain-Driven Design (DDD) principles and the CQRS (Command Query Responsibility Segregation) pattern. This combination ensures a clear separation of concerns, maximum testability and low coupling between layers.

The Dependency Rule

All dependencies point inward. The Domain knows about no other layer — it defines only interfaces (abstractions). ORM, Application and WebApi implement or consume these interfaces. IoC is the only project authorized to "see" multiple layers in order to register the concrete implementations.

CQRS via MediatR

Every operation is represented by a separate Command (write) or Query (read). MediatR automatically routes to the corresponding Handler. This eliminates fat controllers and centralizes the logic in the application handlers.

Data Access Patterns
Repository Pattern and Unit of Work
Abstractions in Domain · Implementations in ORM · AsNoTracking · CommitAsync

Interfaces in Domain (no EF dependency)

Every repository has its interface defined in the Domain project, ensuring the upper layers (Application) never depend on EF Core directly:

// Domain/Repositories/ISaleRepository.cs
public interface ISaleRepository
{
    Task<Sale> CreateAsync(Sale sale, CancellationToken ct = default);
    Task<Sale?> GetByIdAsync(Guid id, CancellationToken ct = default);
    Task<Sale> UpdateAsync(Sale sale, CancellationToken ct = default);
    Task<bool> DeleteAsync(Guid id, CancellationToken ct = default);
    IQueryable<Sale> GetAllQueryable();
}

IQueryable for Pagination and Filters

The GetAllQueryable() method returns IQueryable<T> instead of IEnumerable<T> or List<T>. This allows the GetAll handlers to apply filters, ordering and pagination directly in the database via the SQL generated by EF Core — avoiding loading unnecessary records into memory.

// ORM/Repositories/SaleRepository.cs
public IQueryable<Sale> GetAllQueryable()
    => _context.Sales.AsNoTracking()
                     .Include(s => s.Items)
                     .AsQueryable();
💡
AsNoTracking on all reads
AsNoTracking() is applied on all read queries. EF Core does not need to track entities that will not be modified, resulting in lower memory usage and queries that are 20–40% faster on read-heavy workloads.

IUnitOfWork — Transactional Atomicity

The Unit of Work pattern ensures that multiple write operations are committed atomically. Every write handler calls _unitOfWork.CommitAsync() at the end and never calls SaveChangesAsync() directly on the repository.

// Domain/Repositories/IUnitOfWork.cs
public interface IUnitOfWork
{
    Task<int> CommitAsync(CancellationToken cancellationToken = default);
}

// ORM/Persistence/UnitOfWork.cs
public sealed class UnitOfWork : IUnitOfWork
{
    private readonly DefaultContext _context;
    public UnitOfWork(DefaultContext context) => _context = context;

    public Task<int> CommitAsync(CancellationToken ct = default)
        => _context.SaveChangesAsync(ct);
}
// Application/Sales/CreateSale/CreateSaleHandler.cs
public async Task<CreateSaleResult> Handle(CreateSaleCommand cmd, CancellationToken ct)
{
    var sale = _mapper.Map<Sale>(cmd);
    sale.CalculateTotals();

    await _saleRepository.CreateAsync(sale, ct);
    await _unitOfWork.CommitAsync(ct);     // ← single commit point

    await _eventPublisher.PublishAsync(new SaleCreatedEvent(sale.Id));
    return _mapper.Map<CreateSaleResult>(sale);
}

Decorator Chain: Three Composed Implementations

The Decorator pattern stacks three behaviors around each event publication, without inheritance and without changing the existing classes:

// Decorator chain (configured in IoC):
// MongoEventPublisher
//   └─ LoggingEventPublisher
//        └─ RebusEventPublisher   ← final implementation (sends to RabbitMQ via Rebus)

public sealed class MongoEventPublisher : IEventPublisher
{
    private readonly IEventPublisher _inner;   // LoggingEventPublisher
    private readonly IMongoCollection<DomainEventDocument> _collection;

    public async Task PublishAsync<TEvent>(TEvent @event, CancellationToken ct)
    {
        // 1. Persist to MongoDB (best-effort — does not block on failure)
        try { await _collection.InsertOneAsync(...); } catch { /* log only */ }

        // 2. Delegate to the next decorator
        await _inner.PublishAsync(@event, ct);
    }
}

public sealed class LoggingEventPublisher : IEventPublisher
{
    private readonly IEventPublisher _inner;   // RebusEventPublisher
    private readonly ILogger _logger;

    public async Task PublishAsync<TEvent>(TEvent @event, CancellationToken ct)
    {
        _logger.LogInformation("Publishing {EventType}", typeof(TEvent).Name);
        await _inner.PublishAsync(@event, ct);  // 3. Publish to the bus via Rebus
    }
}

public sealed class RebusEventPublisher : IEventPublisher
{
    private readonly IBus _bus;
    public async Task PublishAsync<TEvent>(TEvent @event, CancellationToken ct)
        => await _bus.Publish(@event);
}

Rebus + RabbitMQ

Rebus is the messaging framework that provides an abstraction over the transport. In production it uses RabbitMQ 3.13; in development and tests it uses the InMemory transport, removing the need for a running RabbitMQ in the unit tests.

// IoC — conditional transport configuration
if (env.IsDevelopment() || env.IsEnvironment("Testing"))
    rebus.Transport(t => t.UseInMemoryTransport(network, "developer-store"));
else
    rebus.Transport(t => t.UseRabbitMq(connectionString, "developer-store"));

MongoDB Event Store

All domain events are persisted to MongoDB as a DomainEventDocument before being sent to the bus. The operation is best-effort: a MongoDB failure does not prevent publication to RabbitMQ. This creates an immutable audit trail of the domain events:

public record DomainEventDocument
{
    public ObjectId Id { get; init; }
    public string EventType { get; init; }
    public string Payload { get; init; }    // serialized JSON
    public DateTime OccurredAt { get; init; }
}

Retry with Polly

Event publication has a retry policy configured with Polly: 3 attempts with exponential backoff (1s → 2s → 4s) on a transient network or broker failure:

Policy
  .Handle<Exception>()
  .WaitAndRetryAsync(3, attempt => TimeSpan.FromSeconds(Math.Pow(2, attempt - 1)))

Identity and Access Control
Authentication and Security
JWT HS256 · BCrypt · OWASP A01 · Role-Based Authorization

JWT HS256 with Strict Validation

Authentication uses JWT with the HS256 (HMAC-SHA256) algorithm. The secret key is validated at application startup — keys shorter than 32 bytes throw an exception and prevent startup:

// Startup validation
if (jwtKey.Length < 32)
    throw new InvalidOperationException(
        "JWT secret key must be at least 32 characters for HS256 security.");

Tokens include validation of Issuer, Audience, Expiration and Signature. Any invalid token results in HTTP 401 before reaching the controller.

BCrypt for Passwords

Passwords are never stored in plain text. The hash is generated with BCrypt (work factor 12), which includes an automatic salt and is resistant to brute-force and rainbow-table attacks:

// Hash when creating a user
user.Password = BCrypt.Net.BCrypt.HashPassword(command.Password);

// Verification on login
BCrypt.Net.BCrypt.Verify(command.Password, user.Password)

OWASP A01 — Ownership Check

The PUT /api/users/{id} endpoint implements object-level access control, addressing one of the most critical vulnerabilities in the OWASP Top 10 (A01 — Broken Access Control). A user can only update their own profile:

// UsersController.cs
[HttpPut("{id}")]
[Authorize]
public async Task<IActionResult> UpdateUser(Guid id, UpdateUserRequest request)
{
    var authenticatedUserId = Guid.Parse(
        User.FindFirstValue(ClaimTypes.NameIdentifier)!);

    if (authenticatedUserId != id)
        return Forbid(); // HTTP 403 — not the resource owner

    // ...
}

Role-Based Authorization (RBAC)

The user role-change endpoint requires the Admin role:

// Only administrators can change roles
[HttpPatch("{id}/role")]
[Authorize(Roles = "Admin")]
public async Task<IActionResult> UpdateUserRole(Guid id, UpdateUserRoleRequest request)
{
    // ...
}

DependencyResolver — Entry Point

// IoC/DependencyResolver.cs
public static class DependencyResolver
{
    public static IServiceCollection AddDeveloperStore(
        this IServiceCollection services,
        IConfiguration config,
        IHostEnvironment env)
    {
        var initializers = new IModuleInitializer[]
        {
            new ApplicationModuleInitializer(),
            new OrmModuleInitializer(),
            new WebApiModuleInitializer()
        };

        foreach (var initializer in initializers)
            initializer.Initialize(services, config);

        return services;
    }
}
// WebApi/Program.cs — extremely clean startup

Quantity-Based Discount Calculation

The Sale entity implements automatic discount rules applied item by item at creation and update time:

Item quantityDiscount applied
1 to 3 units0% — no discount
4 to 9 units10% on the item subtotal
10 to 20 units20% on the item subtotal
More than 20 unitsForbidden — ValidationException thrown
// Domain/Entities/SaleItem.cs
public void ApplyDiscount()
{
    Discount = Quantity switch
    {
        >= 10 and <= 20 => 0.20m,
        >= 4            => 0.10m,
        _               => 0.00m
    };

    if (Quantity > 20)
        throw new DomainException("Cannot sell more than 20 identical items.");

    TotalAmount = UnitPrice * Quantity * (1 - Discount);
}

Sale and Item Cancellation

Sales and individual items can be cancelled. Cancellation is represented via domain events, ensuring full traceability:

// Cancel the whole sale → SaleCancelledEvent
// Cancel a specific item → ItemCancelledEvent (item stays in the list, IsCancelled = true)
sale.CancelItem(itemId);   // validates existence, sets IsCancelled, recalculates the total

Total Consistency

The CalculateTotals() method recalculates the TotalAmount of the whole sale by summing only non-cancelled items. It is called on every write operation before the commit:

public void CalculateTotals()
{
    foreach (var item in Items.Where(i => !i.IsCancelled))
        item.ApplyDiscount();

    TotalAmount = Items
        .Where(i => !i.IsCancelled)
        .Sum(i => i.TotalAmount);
}

Common Questions
Frequently Asked Questions (FAQ)
Answers to the main questions about the project's choices

❓ Why Rebus instead of using RabbitMQ.Client directly?

Rebus provides a transport abstraction — the same publishing code works with InMemoryTransport (tests), RabbitMQ (production) or any other supported backend. Switching brokers requires only a configuration change, not a code change.

❓ Why MongoDB for events and not a PostgreSQL table?

Domain events are heterogeneous documents — each type has a different payload. MongoDB accommodates this naturally with a flexible schema. In addition, MongoDB is an explicit requirement of the challenge as a second database.

❓ Is Redis required?

No. Redis was never a requirement of the challenge specification. The docker-compose provisions only PostgreSQL, MongoDB and RabbitMQ. The README was updated to remove any earlier, incorrect mention of Redis.

❓ Why Testcontainers instead of a dedicated test database?

Testcontainers removes environment dependencies — any machine with Docker can run the tests. The PostgreSQL database starts on demand and is destroyed after the tests, leaving no residual state and requiring no manual setup.

❓ Why ICollectionFixture instead of IClassFixture?

With IClassFixture, each test class starts and destroys its own container — 41 start/stop cycles. With ICollectionFixture, a single container serves the entire integration test collection, drastically reducing the total time.

❓ Why the Decorator instead of a single EventPublisher with multiple responsibilities?

SRP (Single Responsibility Principle). MongoEventPublisher only persists to MongoDB; LoggingEventPublisher only logs; RebusEventPublisher only publishes to the bus. Adding or removing a behavior (e.g. adding an event cache) requires creating a decorator, not modifying existing code (OCP).

❓ How does the OWASP A01 ownership check work?

The JWT's NameIdentifier claim is extracted by the authentication middleware and compared with the route id ({id}). If they differ, it returns 403 Forbidden before running any business logic. This prevents user A from changing user B's data even with a valid token.

❓ Does the 93% coverage include all projects?

No — intentionally. The scope covers Domain and Application, which concentrate 100% of the business logic. ORM (EF Core repositories) and WebApi (controllers) are covered by the integration and functional tests, which run against a real database.
